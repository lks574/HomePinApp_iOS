# 2026-06-12 레시피 모델

레시피 **저장 + 재고 연동** 도메인의 데이터 모델. 식재료=Item 통합
([[2026-06-12-식재료-모델-Item-통합]]) 전제.

## 결정

- `Recipe` `@Model` + `RecipeIngredient` `@Model` + `RecipeStep`(**Codable 값**).
- **재료 라인 = `RecipeIngredient` @Model** — `item: Item?` 관계로 재고 연동.
- **단계 = `Recipe.steps: [RecipeStep]`** (Codable 값 배열, 순서=배열 순서).
- **태그 = 기존 [[Tag]] 재사용**(N:N) — `Tag` 에 `recipes` 인버스 추가.
- **[[Item]] 에 `usedInIngredients` 인버스 추가**(`.nullify`) — 역참조·임박 추천용.

## 재고 매칭 / 보유 판정

재료를 재고(`Item`)와 맞추는 **2단 전략**:

1. **명시적 링크 우선** — `RecipeIngredient.item` 이 설정돼 있으면 그 Item 기준.
   AI 파싱/등록 시(또는 사용자가) 재료를 특정 Item 에 연결한다(있으면 매핑, 없으면
   "재고에 추가?" 제안 — [[2026-06-12-식재료-모델-Item-통합]]).
2. **미링크 시 이름 매칭** — `Item.normalizedName`(정규화 키: 소문자·공백제거·동의어)
   ↔ 재료 정규화명. 작은 **동의어 사전 + AI 정규화**로 "배추김치=김치" 흡수. 한 재료가
   여러 Item 에 매칭되면 합산/존재로 본다(김치냉장고+냉장고의 김치).

**"부족" 의 정의** — 정밀 무게 부족(200g 중 150g)은 **하지 않는다**(집 재고 무게를
정밀 입력하는 건 비현실적 + 단위 환산 부담). 대신 **"레시피 재료 중 재고에 *없는*
것"** 을 부족으로 본다.

- 재료별 상태: `보유`(매칭 Item 있고 수량>0) / `없음`(매칭 없음 또는 0).
- 레시피별 요약: 없는 재료 수("N개 부족"). 예: 김치찌개 → 돼지고기·두부 보유,
  **김치 없음**, 대파 없음 → "2개 부족". 없는 재료 0이면 **"지금 가능"**.
- **임박 소비 추천**: `expiresAt` 임박 `Item` → `usedInIngredients` → 그 `Recipe` 들.

## 삭제 규칙

- `Recipe` 삭제 → `ingredients` **cascade**(라인 함께 삭제). `steps` 는 값이라 함께.
- `Item` 삭제 → `RecipeIngredient.item` **nullify**(레시피는 `name` 유지, 링크만 끊김).
- `Tag` 삭제 → `items`/`recipes` nullify(M:N 기본).

## 미정 / 후속

- **정밀 수량 부족**(200g 중 150g): 기본 안 함. 정말 필요하면 후속에 `Item` 에
  measured amount+unit 추가 + 단위 환산 도입(복잡 → 보류 가능성 높음).
- **동의어 사전** 규모·관리(AI 보조 vs 수동 시드).
- `cuisine` 문자열 → 추후 enum/별도 모델 승격 여부.
- 구현 시: `AppModelContainer.models` 에 `Recipe`·`RecipeIngredient` 등록(`RecipeStep`
  은 값이라 불필요). `Item` 에 `normalizedName`(indexed)·`usedInIngredients`(.nullify),
  `Tag` 에 `recipes` 추가.
